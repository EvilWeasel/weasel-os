# WeaselOS Modularization Migration Plan

This plan describes how to move the repository from the current state to the target architecture defined in [target-architecture.md](./target-architecture.md).

The plan is intentionally detailed enough that another agent could continue the work.

## Scope

### Primary goals

- reduce duplication between laptop and desktop
- make the repository understandable and extensible
- preserve the current laptop-first behavior
- keep the current `nixy-*` host naming
- prepare the repo so another user can fork it and bootstrap their own host
- add complete documentation for first-time setup, migration, rebuild, and update flow

### Constraints

- the laptop configuration must remain functional after refactors
- DMS-generated Niri files must remain mutable
- no accidental reintroduction of Hyprland as the main path

## Current State Summary

### Structural pain points

- `flake.nix` manually wires two hosts with near-duplicate Home Manager setup.
- `hosts/nixy-desktop/config.nix` and `hosts/nixy-laptop/config.nix` are both large monoliths.
- `hosts/nixy-desktop/home.nix` and `hosts/nixy-laptop/home.nix` also contain heavy duplication.
- `modules/` mixes real reusable modules with inherited historical content.
- `home-modules/` is not a coherent layer yet.

### Portability blockers

- `flake.nix` uses absolute local paths for flake inputs:
  - `path:/home/evilweasel/weasel-os/packages/helium`
  - `path:/home/evilweasel/weasel-os/packages/t3code`
- The repo assumes clone path `~/weasel-os` in aliases.
- Some files still hardcode personal usernames or placeholders.
- The README is still Hyprland-centric and not adequate for onboarding.

### Niri/DMS-specific blockers

Observed live structure in `~/.config/niri`:

- `config.kdl`
- `base/animations.kdl`
- `base/binds.kdl`
- `base/input.kdl`
- `base/layout.kdl`
- `base/monitors.kdl`
- `base/windowrules.kdl`
- `dms/*.kdl`
- `dms/profiles/*.kdl`

Current repo state:

- `programs/niri.nix` writes `config.kdl`
- `config.kdl` includes `base/*.kdl` and `dms/*.kdl`
- the live `base/*.kdl` files are not yet stored in the repo
- the live `dms/*.kdl` files are generated by DMS and should not be declaratively owned

Implication:

- a fresh install does not yet have the required repo-owned Niri base layer
- a fresh install may also not yet have DMS-generated files such as `dms/outputs.kdl`
- current flake behavior is therefore not robust enough for another user bootstrap

## Resolved Assumptions

- Keep `programs/`, `packages/`, and `scripts/`.
- Keep `nixy-*` host naming.
- Keep Niri plus DMS as the primary desktop stack.
- Treat the laptop configuration as the compatibility baseline.
- DMS `dms/*.kdl` files remain mutable and DMS-owned.

## Migration Strategy

Use incremental phases. Do not attempt a giant one-shot rewrite.

### Phase 0: Protect the current working state

Goals:

- establish a safe baseline
- avoid losing the currently working laptop setup

Tasks:

1. Keep the current branch green with:
   - `nix eval --no-write-lock-file .#nixosConfigurations.nixy-laptop.config.system.build.toplevel.drvPath`
   - `nix eval --no-write-lock-file .#nixosConfigurations.nixy-desktop.config.system.build.toplevel.drvPath`
2. Treat the current docs in `docs/` as the planning memory instead of relying on turn context.
3. Do not move multiple layers at once without validation between steps.

Validation:

- both host evals succeed before each major phase begins

### Phase 1: Remove portability blockers first

This phase should happen before the large structural refactor.

Tasks:

1. Replace absolute flake package paths in `flake.nix` with repo-relative paths.
2. Audit and fix hardcoded username or placeholder values:
   - `configHome = "/home/yourusername"`
   - any absolute `/home/evilweasel/...` assumptions
3. Decide how strict the repo path convention should be:
   - preferred: support any clone path
   - acceptable fallback: recommend `~/weasel-os`, but do not hard-break elsewhere
4. Update local and documented repo naming to `WeaselOS` / `weasel-os`.

Validation:

- both host evals still succeed
- fresh-user-facing instructions no longer depend on `evilweasel`

### Phase 2: Introduce the target scaffolding

Create new structure without moving everything at once.

Tasks:

1. Add `lib/` with host composition helpers.
2. Add `profiles/system/` and `profiles/home/`.
3. Keep old host files intact while the new layers are introduced.
4. Update `flake.nix` so host generation can gradually move to helper-based composition.

Implementation note:

- prefer additive migration first
- do not delete old modules until the new composition is proven

Validation:

- both hosts still evaluate
- `flake.nix` becomes simpler, not more bespoke

### Phase 3: Extract shared NixOS system composition

Move duplicated system policy into profiles.

Expected profile candidates:

- `profiles/system/base.nix`
- `profiles/system/niri-dms.nix`
- `profiles/system/virtualization.nix`
- `profiles/system/gaming.nix`
- `profiles/system/desktop-common.nix`
- `profiles/system/laptop-common.nix`

Tasks:

1. Identify duplicated sections from both host `config.nix` files.
2. Extract common locale, fonts, shell tooling, common services, audio, printing, SSH, flatpak, virtualization, and package groups.
3. Leave only genuine host-specific deltas in `hosts/<host>/`.
4. Move one concern at a time, validating after each move.

Validation:

- desktop eval
- laptop eval
- optional syntax parse on touched files

### Phase 4: Extract shared Home Manager composition

Move duplicated home configuration into home profiles.

Expected profile candidates:

- `profiles/home/base.nix`
- `profiles/home/shell.nix`
- `profiles/home/git.nix`
- `profiles/home/desktop-common.nix`
- `profiles/home/niri-dms.nix`
- `profiles/home/dev-tools.nix`

Tasks:

1. Extract shell aliases, Git setup, Kitty, shell init, common packages, Hyprlock settings, and shared user experience config.
2. Keep app-specific modules in `programs/`.
3. Leave only host-local deltas in the host home files.

Validation:

- laptop eval after each extraction step
- desktop eval after each extraction step

### Phase 5: Normalize module boundaries

Reorganize `modules/` to reflect intent.

Tasks:

1. Create `modules/nixos/` and `modules/home/`.
2. Move driver modules into `modules/nixos/drivers/`.
3. Move feature toggles such as CAN bus into `modules/nixos/features/`.
4. Move service modules such as `llama-cpp` into service namespaces.
5. Move `home-modules/llama-cpp.nix` into `modules/home/services/llama-cpp.nix`.
6. Review Apple Silicon and bundled firmware content:
   - keep only if WeaselOS still intends to support it
   - otherwise remove from the active core and document the removal

Validation:

- no functional change beyond import path updates
- both host evals stay green

### Phase 6: Fix the Niri/DMS ownership model

This phase is mandatory for bootstrap compatibility.

Tasks:

1. Bring the live `~/.config/niri/base/*.kdl` files into the repo.
2. Store them under a repo-managed path, for example:
   - `programs/niri/base/animations.kdl`
   - `programs/niri/base/binds.kdl`
   - `programs/niri/base/input.kdl`
   - `programs/niri/base/layout.kdl`
   - `programs/niri/base/windowrules.kdl`
3. Make Home Manager install those base files declaratively.
4. Keep `~/.config/niri/config.kdl` declarative as the entrypoint.
5. Do not declaratively own `~/.config/niri/dms/*.kdl`.
6. Add a bootstrap mechanism for missing DMS files on first login:
   - create mutable files only if absent
   - do not create store symlinks for DMS-owned files
7. Fix DMS greeter `configHome` to derive from the actual user home.

Recommended implementation direction:

- manage `config.kdl` plus `base/*.kdl` in-repo
- use a small activation/bootstrap step for initial `dms/*.kdl` presence if needed
- verify behavior on a clean user profile, not only on your existing machine

Important caution:

- `dms/outputs.kdl` is generated by DMS based on monitor settings and must stay mutable
- `dms/profiles/*.kdl` also remains DMS-owned

Validation:

- fresh user profile should reach a working Niri session
- after reboot, system still works
- DMS can still modify monitor/output data through its GUI

### Phase 7: Simplify host creation for forks

Make it realistic for another user to add a new host with minimal edits.

Tasks:

1. Define what a new host must contain:
   - `hardware.nix`
   - `users.nix`
   - `variables.nix`
   - minimal host metadata
2. Ensure a new host does not require copy-pasting a 600-line config file.
3. Optionally add a host template directory or documented host skeleton.

Validation:

- a new host can be created by following the README only

### Phase 8: Rewrite README for real onboarding

This is not an afterthought. It is part of the deliverable.

The README should include:

1. Project overview
2. Supported desktop direction: Niri + DMS
3. Repo layout overview
4. Fresh install instructions
5. Existing NixOS migration instructions
6. New host creation instructions
7. First rebuild command
8. Post-switch commands:
   - `fr`
   - `fu`
9. Troubleshooting and rollback notes

The first-time switch command should be:

```bash
sudo nixos-rebuild switch --flake ~/weasel-os#<host>
```

Reason:

- `fr` and `fu` are not available until after the configuration is active
- `nh` may not be available yet on a fresh system

README must also explain that `fr` and `fu` only become available after the first successful switch and re-login.

### Phase 9: Remove obsolete leftovers

Only after the new structure is working.

Tasks:

1. Remove clearly obsolete Hyprland-only code if it no longer serves a purpose.
2. Remove stale comments and dead imports.
3. Remove unused ZanyOS leftovers that are no longer part of the supported design.

Validation:

- both host evals still pass after cleanup

## Risk Register

### Highest risk: losing working state during a huge refactor

Mitigation:

- use the docs as the source of truth
- refactor by phases
- validate after each phase

### High risk: DMS mutable-vs-declarative ownership conflict

Mitigation:

- repo owns `config.kdl` and `base/*.kdl`
- DMS owns `dms/*.kdl`
- bootstrap missing mutable files without turning them into store symlinks

### High risk: hidden local state masks missing repo content

Mitigation:

- compare repo-managed files against live `~/.config/niri`
- test against a clean user profile or equivalent bootstrap path

### Medium risk: path portability remains broken

Mitigation:

- remove absolute repo paths first
- document expected clone path clearly if any preference remains

## Validation Matrix

Use these commands repeatedly throughout the migration:

```bash
nix-instantiate --parse <changed-file>
nix eval --no-write-lock-file .#nixosConfigurations.nixy-laptop.config.system.build.toplevel.drvPath
nix eval --no-write-lock-file .#nixosConfigurations.nixy-desktop.config.system.build.toplevel.drvPath
```

Additional validation to add once onboarding is implemented:

- clean-user bootstrap test
- first-login DMS/Niri startup test
- reboot test
- alias availability test for `fr` and `fu`

## Definition of Done

The migration is done when all of the following are true:

- shared system/home composition is extracted into reusable profiles
- host files are small and host-specific
- no absolute personal repo paths remain
- Niri base layer is in-repo
- DMS mutable files are intentionally handled
- README can bootstrap a fresh user from fork to working switch
- `fr` and `fu` are documented and work after first activation
- laptop and desktop both evaluate successfully
